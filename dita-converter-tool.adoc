== Overview

The DITA Converter Tool is a lightweight, open-source proof-of-concept application that converts text-based *PDF* and *DOCX* files into valid *DITA 1.3 XML* via a rule-based pipeline.

The tool was built to demonstrate the viability of automated structured content conversion — taking unstructured technical documentation and producing DITA-compliant XML output that can be consumed by any DITA-aware content management system or publishing pipeline.

.Key facts
[cols="1,2"]
|===
|DITA Version |1.3 (2.0 migration path documented)
|Conversion Time |~2.3 seconds per document
|Input Formats |Text-based PDF, Microsoft Word DOCX
|Output Format |DITA 1.3 XML (concept, task, reference)
|UI |Streamlit web application
|License |MIT
|===

== Pipeline Architecture

The tool processes documents through a sequential four-module pipeline. Each module has a single, well-defined responsibility and communicates through a shared internal data structure called the *Content Tree*.

[source]
----
Upload PDF or DOCX
        │
        ▼
  ┌─────────────┐
  │  Extractor  │  pdfplumber / python-docx
  └─────────────┘  PDF/DOCX → Content Tree (list of block dicts)
        │
        ▼
  ┌─────────────┐
  │   Mapper    │  config/mapping_rules.yaml (PyYAML)
  └─────────────┘  Content Tree → Annotated Content Tree
        │
        ▼
  ┌─────────────┐
  │  Generator  │  lxml
  └─────────────┘  Annotated Tree → DITA 1.3 XML string
        │
        ▼
  ┌─────────────┐
  │  Validator  │  lxml well-formedness check
  └─────────────┘  XML → ValidationResult + human-readable report
        │
        ▼
  Download .dita + View report in UI
----

=== Content Tree Block Schema

The Content Tree is the internal data structure passed between pipeline modules. Each block represents one semantic unit from the source document.

[cols="1,1,3"]
|===
|Field |Type |Description

|`type`
|`str`
|Block category: `heading`, `paragraph`, `list_item`, `table`, `figure`, `code_block`

|`level`
|`int \| None`
|Heading depth (1–6). `None` for non-headings

|`text`
|`str`
|Raw extracted text content

|`style_name`
|`str \| None`
|Original DOCX style name (e.g. `Heading 1`, `Body Text`). `None` for PDF blocks

|`dita_element`
|`str \| None`
|Assigned DITA element after mapping (e.g. `title`, `p`, `note:warning`). `None` before mapping

|`children`
|`list \| None`
|Nested blocks for tables and lists

|`attributes`
|`dict \| None`
|Extra metadata (e.g. `{"is_bold": true, "note_type": "warning"}`)
|===

== Mapping Rules

The `config/mapping_rules.yaml` file controls how source document elements map to DITA 1.3 elements. *No code changes are required* to adapt the tool to a new document style — only the YAML file needs updating.

=== Heading Map

[source,yaml]
----
heading_map:
  h1_first: title          # First H1 → topic <title>
  h1: section_title        # Subsequent H1 → <section><title>
  h2: sectiondiv_title     # H2 → <sectiondiv><title>
  h3: sectiondiv_title     # H3 → nested <sectiondiv><title>
----

=== Note / Callout Map

Callout boxes are detected by pattern matching on block header text and mapped to the appropriate DITA `<note>` type.

[source,yaml]
----
note_map:
  - pattern: "IMPORTANT INFORMATION"
    element: note
    type: important
  - pattern: "^WARNING"
    element: note
    type: warning
  - pattern: "^CAUTION"
    element: note
    type: caution
  - pattern: "^DANGER"
    element: note
    type: danger
  - pattern: "^Note[s]?:"
    element: note
    type: note
----

=== Table Map

[source,yaml]
----
table_map:
  default: table           # Full CALS structure with <thead> + <colspec>
  abbreviations: dl        # Term/Description 2-col tables → <dl><dlentry>
  abbreviation_signals:
    - "Term"
    - "Abbreviation"
    - "Acronym"
----

== Installation

=== Prerequisites

* Python 3.9 or later
* pip

=== Steps

. Clone the repository:
+
[source,bash]
----
git clone https://github.com/your-org/dita-converter.git
cd dita-converter
----

. Install dependencies:
+
[source,bash]
----
pip install -r requirements.txt
----

. Run the UI:
+
[source,bash]
----
streamlit run ui/app.py
----

. Open your browser at `http://localhost:8501`

== Development Agent Model

This project was built using a structured four-agent development workflow. These are roles played during the development process — they are not components of the runtime tool.

[cols="1,1,3"]
|===
|Agent |Role |Responsibilities

|*Allocator*
|Task Planning
|Defines tasks, acceptance criteria, and sequencing for each build session

|*Coder*
|Implementation
|Writes all production code, unit tests, and configuration files

|*Reviewer*
|Quality Assurance
|Reviews code for correctness, edge cases, and DITA spec adherence

|*Scribe*
|Documentation
|Writes commit messages, updates docs, and maintains the changelog
|===

Each build session follows a fixed handoff sequence:

[source]
----
ALLOCATOR → defines task + acceptance criteria
    │
    ▼
CODER → implements the task
    │
    ▼
REVIEWER → validates against criteria
    │
    ▼
SCRIBE → commits + documents the approved change
----

No code is committed without passing the Reviewer gate. No task begins without Allocator-defined criteria.
